# Правила работы с репозиторием ZVONA

## Безопасность данных

- Использовать только выдуманные демонстрационные данные.
- Не добавлять реальные телефоны, имена клиентов, клиентские базы и персональные данные.
- Не добавлять в репозиторий API-ключи, токены, пароли, `.env` и другие секреты.
- Не подключать внешние API и реальные интеграции без отдельного запроса владельца проекта.

## Архитектура

- Приложение построено на Next.js App Router и TypeScript. UI должен работать через серверные API, а не имитировать сохранение только в локальном состоянии.
- Основное хранилище — PostgreSQL. Изменения схемы оформлять новыми последовательными миграциями; не переписывать уже применённые миграции.
- Локальный и production-запуск выполняются через Docker Compose. В составе пилота есть Next.js app, отдельный TypeScript worker, PostgreSQL и S3-совместимое object storage.
- Аутентификация использует приглашения и серверные сессии. Каждый запрос, repository-запрос, job, audit event, storage path и экспорт обязан быть изолирован по `organization_id`.
- Фоновые задания хранятся в PostgreSQL и должны сохранять retry, idempotency и audit trail.
- OpenAI вызывается только на сервере. Structured Outputs валидируются перед сохранением; неподтверждённые claims не должны попадать в operator task card или outreach draft.
- Email и WhatsApp в MVP остаются только черновиками. Не добавлять автоматическую отправку, Twilio, CRM API, billing, внешний enrichment или неофициальные WhatsApp-интеграции без отдельной задачи.
- Demo fixtures допустимы только как явно обозначенный fallback. Новые продуктовые действия должны проходить через API/repository и сохраняться в PostgreSQL.
- Не усложнять архитектуру без необходимости: Temporal, Redis, vector store и публичный marketplace находятся за пределами пилота.
- Оценивать действия оператора, а не коммерческий результат разговора. Отказ клиента сам по себе не снижает качество.

## Проверка изменений

- Для бизнес-правил добавлять unit-тесты; для auth, tenant isolation, jobs и API — integration/contract-тесты со scoped fixtures.
- Перед завершением изменения запускать релевантные тесты, TypeScript/build и, когда затронут runtime, Docker smoke test.
- Для safe fetch проверять DNS/IP до запроса, каждый redirect, фактический peer address, лимит ответа и запрет private/reserved networks.
- Не запускать `npm audit fix --force` и semver-major обновления вслепую. Сначала фиксировать затронутую цепочку и проверять совместимость Next.js.

## Документация и Git

- Обновлять файлы в `docs/` при изменении продукта, предметной модели или правил расчёта.
- Не выполнять `git push` без явного разрешения пользователя.
- Не коммитить сгенерированные сборки, зависимости и локальные настройки.
