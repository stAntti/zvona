# Правила работы с репозиторием ZVONA

## Главный приоритет до первых платящих клиентов

`Product > Engineering.`

Работать как founding engineer стартапа. Главная цель — максимально быстро проверять продуктовые гипотезы. Не добавлять инфраструктуру, безопасность, оптимизации, тесты или архитектурные улучшения, если они не требуются для текущего пользовательского сценария.

Любое изменение должно отвечать на вопрос:

> Как это помогает получить следующего пользователя или следующую продажу?

Если не помогает — сначала запросить разрешение и не начинать работу самостоятельно.

Текущая стадия продукта определяется в `PRODUCT_STAGE.md`. Перед каждой задачей сначала письменно ответить:

> Почему эта задача увеличивает вероятность продажи продукта?

Если прямого и убедительного ответа нет — задачу не выполнять.

Приоритет текущих итераций:

1. Пользовательский сценарий и понятность продукта.
2. Demo, которое можно уверенно показать клиенту.
3. Скорость получения видимого результата.
4. Красивый, цельный интерфейс.

Запрещено самостоятельно начинать performance-работы, security hardening, оптимизацию, масштабирование, dependency cleanup или усложнение архитектуры, если этого прямо не требует текущая пользовательская история.

Ближайшие продуктовые поверхности:

- карточка компании;
- AI summary;
- очередь звонков;
- экран оператора;
- экран QA;
- экран руководителя;
- onboarding;
- demo-режим.

## Безопасность данных

- Использовать только выдуманные демонстрационные данные.
- Не добавлять реальные телефоны, имена клиентов, клиентские базы и персональные данные.
- Не добавлять в репозиторий API-ключи, токены, пароли, `.env` и другие секреты.
- Не подключать внешние API и реальные интеграции без отдельного запроса владельца проекта.

## Архитектура

- Сохранять существующий стек Next.js/TypeScript и уже работающие API/PostgreSQL/Docker-компоненты, но не расширять инфраструктуру без прямой продуктовой необходимости.
- Demo-режим является полноценным способом показа продукта, а не аварийным fallback. Он должен запускаться быстро и использовать только выдуманные данные.
- Backend использовать там, где он уже поддерживает демонстрируемый сценарий. Не задерживать UX из-за построения универсального backend.
- OpenAI вызывается только на сервере. Structured Outputs валидируются перед сохранением; неподтверждённые claims не должны попадать в operator task card или outreach draft.
- Email и WhatsApp в MVP остаются только черновиками. Не добавлять автоматическую отправку, Twilio, CRM API, billing, внешний enrichment или неофициальные WhatsApp-интеграции без отдельной задачи.
- Не усложнять архитектуру без необходимости: Temporal, Redis, vector store и публичный marketplace находятся за пределами пилота.
- Оценивать действия оператора, а не коммерческий результат разговора. Отказ клиента сам по себе не снижает качество.

## Проверка изменений

- Проверять в первую очередь полный пользовательский путь и визуальное качество изменённого экрана.
- Запускать только релевантные тесты и production build; не расширять тестовую инфраструктуру без необходимости текущего сценария.
- Не заниматься audit, dependency upgrades и технической чисткой в продуктовой итерации, если они не блокируют запуск demo.

## Документация и Git

- Обновлять файлы в `docs/` при изменении продукта, предметной модели или правил расчёта.
- Не выполнять `git push` без явного разрешения пользователя.
- Не коммитить сгенерированные сборки, зависимости и локальные настройки.
